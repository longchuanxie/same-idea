# 01 阶段 0：领域模型与状态协议

> 统一实体、状态机、权限、副作用和跨模块契约，是后续 LLM 实施的地基。

## 1. 阶段目标

阶段 0 的目标是把产品概念模型转成工程模型。该阶段不追求 UI 完成，而是让后续所有模块共享同一套实体、枚举、状态机和权限规则。

完成后，LLM 在实现编辑器、Agent、导出和存储时，不应再自行发明 Project、Chapter、Revision、Session 等结构。

## 2. 核心实体

| 实体 | 说明 | 关键关系 |
| --- | --- | --- |
| Project | 小说项目根对象 | 包含 Outline、Chapter、Character、Worldbuilding、Note、Session、Snapshot |
| OutlineNode | 大纲节点，可表示卷/章/节/场景 | 可关联 Chapter，可嵌套 |
| Chapter | 正文写作单位 | 属于 Project，可绑定 OutlineNode，拥有 content 与 revisions |
| Revision | 修订记录 | 属于 Chapter，由用户或 Agent 产生，可接受/拒绝 |
| Character | 角色卡 | 属于 Project，可被章节、笔记、关系图引用 |
| SettingEntry | 世界观设定条目 | 属于 Project，可按分类索引 |
| Note | 资料或灵感笔记 | 可来自用户输入、Agent 摘要或搜索结果 |
| Session | Agent 对话会话 | 属于 Project，可绑定阶段、任务、上下文快照 |
| AgentTask | Agent 工具执行实例 | 记录输入、输出、状态、副作用 |
| Snapshot | 项目快照 | 用于回溯、分支或手动备份 |

## 3. 推荐基础模型字段

以下为工程字段建议，具体语言可用 Dart class / freezed / json_serializable 表达。

```text
Project {
  id, title, author, description, language, genre, tags,
  defaultStyleProfileId?, activeChapterId?, createdAt, updatedAt,
  schemaVersion, localEncryptionEnabled
}

Chapter {
  id, projectId, outlineNodeId?, title, order,
  contentFormat: markdown | delta | html,
  content, plainTextCache, wordCount,
  status: draft | reviewing | revised | locked | published,
  createdAt, updatedAt
}

Revision {
  id, projectId, chapterId,
  source: user | agent,
  operation: insert | delete | replace,
  rangeAnchor, beforeText, afterText,
  status: pending | accepted | rejected | superseded,
  prompt?, model?, agentTaskId?, createdAt, resolvedAt?
}
```

## 4. 内容格式协议

| 格式 | 用途 | 要求 |
| --- | --- | --- |
| markdown | 推荐默认正文格式 | 利于 diff、导出、存储和 LLM 处理；富文本能力以扩展标记表达。 |
| delta | 后续增强富文本编辑 | 适合复杂富文本，但需要导出层适配。 |
| html | 导入/导出中间格式 | 不建议作为主存储格式，避免清洗复杂。 |

## 5. 章节状态机

章节状态用于 UI、导出和 Agent 决策。

```text
draft -> reviewing -> revised -> published
  |          |            |
  +----------+------------+-> locked
```

规则：
- draft：可自由编辑。
- reviewing：存在待处理修订。
- revised：修订已处理但尚未发布。
- published：导出或标记完成，可继续解锁编辑。
- locked：临时锁定，防止 Agent 任务执行期间冲突写入。

## 6. Agent 权限矩阵

| 操作 | 默认权限 | 用户确认 | 备注 |
| --- | --- | --- | --- |
| 生成建议 | 允许 | 不需要 | 只显示建议，不写入项目。 |
| 插入正文 | 受限 | 需要 | 生成内容进入 pending revision 或草稿区。 |
| 替换选区 | 受限 | 需要 | 必须保留 beforeText。 |
| 删除正文 | 高风险 | 必须 | 不能静默删除，只能生成删除修订。 |
| 创建笔记 | 允许 | 可选 | 搜索摘要可自动入资料箱，但需可撤销。 |
| 修改角色/设定 | 受限 | 需要 | 以建议形式呈现。 |
| 联网搜索 | 可配置 | 首次需要 | 显示查询词、来源和缓存状态。 |
| 导出文件 | 允许 | 需要选择路径/格式 | 不改变项目正文。 |

## 7. 跨模块事件

建议定义领域事件，便于解耦 UI、Agent、存储和索引：

```text
ProjectCreated
ChapterContentChanged
RevisionCreated
RevisionAccepted
RevisionRejected
CharacterUpdated
NoteCreated
AgentTaskStarted
AgentTaskCompleted
AgentTaskFailed
SnapshotCreated
ExportCompleted
```

事件应只描述事实，不直接执行业务逻辑。

## 8. 阶段验收

| 验收项 | 通过标准 |
| --- | --- |
| 实体定义 | 所有核心实体有 id、projectId、createdAt、updatedAt、schemaVersion 或迁移策略。 |
| 状态机 | 章节、修订、AgentTask 均有明确状态和转换规则。 |
| 权限矩阵 | 每个会写入项目数据的 Agent 操作都有确认规则。 |
| 序列化 | 实体可 JSON 序列化和反序列化，字段缺失有默认值或迁移逻辑。 |
| 示例数据 | 提供一个包含项目、章节、角色、笔记、会话、修订的完整样例。 |
