# 00 需求总纲与实施原则

> 定义产品目标、用户价值、LLM 实施方式、全局边界和验收方法。

## 1. 文档目的

本文档将 Novel Creator — LLM 驱动的小说创作 Agent 从产品蓝图改写为可由 LLM 按阶段实施的需求设计规格。这里的“阶段”不是传统版本规划，而是工程依赖层与验收切片：所有目标功能可以在总架构中一次性纳入设计，但必须按可运行闭环逐层生成、测试、集成和回滚。

核心原则：
- 全量规划：一次性确定产品目标、领域模型、交互框架和扩展方向。
- 分层实施：按照依赖关系推进，不按“简化版/完整版”区分产品价值。
- 连续验收：每一阶段都有可运行产物、验收用例和失败降级。
- 用户可控：Agent 是协作者，不能绕过用户确认直接破坏正文、设定或项目资产。
- 本地优先：默认所有创作资产保存在本地，联网与模型调用必须透明可控。

## 2. 产品定位

Novel Creator 是面向长篇小说、网文和长文虚构创作的本地优先 AI 写作工作台。它不是“聊天生成器”，而是把正文编辑、项目知识库、Agent 协作、修订审核、资料调研和成稿导出整合在一个创作空间中。

推荐定位表述：
> 一个本地优先的 AI 小说创作工作台，帮助作者管理大纲、角色、世界观、章节正文和调研资料，并通过可审核的 Agent 修订机制完成续写、改写、润色和一致性检查。

## 3. 目标用户与核心任务

| 用户类型 | 高频任务 | 关键体验要求 |
| --- | --- | --- |
| 网文作者 | 快速构思、分章写作、日更续写、节奏检查 | 低摩擦、高产出、章节连贯、字数统计 |
| 业余创作者 | 从想法生成大纲、完善角色、慢速写作 | 易上手、少配置、能解释 AI 建议 |
| 专业作者 | 资料管理、修订审核、版本对比、导出投稿 | 控制感、可追踪、可回滚、导出质量稳定 |
| LLM 实施者 | 按规格生成模块、运行测试、修复集成问题 | 接口清楚、数据结构稳定、验收标准明确 |

## 4. 全局用户闭环

最重要的产品闭环如下：

1. 用户创建小说项目，填写标题、类型、语言、目标风格。
2. 用户输入故事创意，Agent 生成多个方向或大纲建议。
3. 用户选定方向，生成章节结构、角色卡和世界观初稿。
4. 用户打开某一章，在编辑器中写作或让 Agent 续写。
5. Agent 的正文修改以修订形式进入编辑器，用户接受、拒绝或二次改写。
6. 用户随时将资料、搜索结果、设定写入项目知识库。
7. Agent 基于项目知识库检查角色、时间线、设定和风格一致性。
8. 用户选择格式导出成稿。

任何功能都应服务于这条闭环；不能服务闭环的功能需要默认隐藏或作为高级能力。

## 5. 阶段划分

| 阶段 | 名称 | 目标产物 | 依赖 |
| --- | --- | --- | --- |
| 阶段 0 | 需求协议与领域模型 | 统一术语、实体、状态机、权限矩阵 | 无 |
| 阶段 1 | 基础工程与本地存储 | Flutter 工程骨架、数据库封装、设置管理 | 阶段 0 |
| 阶段 2 | 项目工作台与编辑闭环 | 项目/章节/编辑器/保存/导出基础闭环 | 阶段 1 |
| 阶段 3 | Agent 基础能力 | LLM Client、对话面板、基础写作工具 | 阶段 1-2 |
| 阶段 4 | 修订追踪与审核 | diff、接受/拒绝、撤销/恢复、修订历史 | 阶段 2-3 |
| 阶段 5 | 结构化小说知识库 | 角色、世界观、笔记、大纲和一致性索引 | 阶段 0-4 |
| 阶段 6 | 联网搜索与引用 | 搜索、提取、资料卡、引用治理、缓存 | 阶段 3、5 |
| 阶段 7 | 高级会话与时光机 | 多 Session、摘要、分支、快照、回溯 | 阶段 3-5 |
| 阶段 8 | 导出与多端体验完善 | EPUB/PDF/DOCX、响应式布局、快捷键 | 阶段 2、4、5 |

## 6. 全局非功能要求

| 类别 | 要求 |
| --- | --- |
| 可靠性 | 任意 Agent 操作失败不能损坏用户正文；所有写入操作应有事务或可恢复记录。 |
| 性能 | 打开项目首屏不加载全部章节正文；长章节应懒加载；流式生成不阻塞编辑器。 |
| 隐私 | API Key 本地加密保存；联网搜索和模型调用前应明确哪些文本会被发送。 |
| 可移植 | 项目应支持导出为单个项目包，便于备份和未来云同步。 |
| 可测试 | 每个 Agent 工具、存储仓库、编辑器命令、导出格式都有验收用例。 |
| 可降级 | 搜索、模型、导出高级格式失败时，基础编辑和本地保存必须继续可用。 |

## 7. LLM 实施规则

给 LLM 生成代码时，应遵守以下规则：

- 不允许一次生成无边界的大模块；每次生成必须声明改动文件、接口和测试方式。
- 数据模型先行，UI 和 Agent 不得私自定义重复模型。
- 所有会修改项目数据的 Agent 工具必须返回结构化结果，由应用层统一写入。
- 编辑器与 Agent 解耦；Agent 不直接操作 UI 控件，只提交 EditorCommand 或 RevisionPatch。
- 所有异步任务必须有状态：pending、running、succeeded、failed、cancelled。
- 所有失败都必须进入可展示错误对象，禁止只打印日志。

## 8. 全局完成定义

一个阶段被认为完成，需要同时满足：

- 功能路径可在 UI 中执行。
- 数据能正确保存、关闭应用后恢复。
- 至少包含正常路径、失败路径、边界输入的测试。
- 文档中定义的实体和接口没有被临时字段绕开。
- 用户可理解当前状态，不会因为 Agent 执行中、失败或取消而丢失内容。
