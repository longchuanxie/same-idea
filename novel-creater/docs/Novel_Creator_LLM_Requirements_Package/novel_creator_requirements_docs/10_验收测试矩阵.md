# 10 验收测试矩阵

> 把各阶段需求转成可执行测试清单，供 LLM 逐项生成与验证。

## 1. 测试原则

验收测试的目标不是覆盖所有代码，而是保证每个阶段的用户闭环、数据完整性和失败降级成立。LLM 每完成一个模块，应补齐或更新对应测试。

测试分三类：
- Unit：实体、Repository、工具函数、diff 算法。
- Integration：AgentTask、存储、编辑器命令、导出流程。
- E2E/Manual：用户从 UI 完成核心路径。

## 2. 核心路径测试

| ID | 路径 | 预期 |
| --- | --- | --- |
| E2E-001 | 新建项目 -> 输入正文 -> 自动保存 -> 重启 | 正文和章节结构完整恢复。 |
| E2E-002 | 创建 3 个章节 -> 调整顺序 -> 导出 Markdown | 导出顺序与 UI 顺序一致。 |
| E2E-003 | 配置模型 -> Agent 续写 -> 生成待处理修订 | 正文不被直接覆盖。 |
| E2E-004 | 接受 Agent 修订 -> 撤销 -> 重做 | 正文和修订状态符合预期。 |
| E2E-005 | 创建角色事实 -> 检查当前章一致性 | 返回至少一个结构化检查结果或无问题说明。 |
| E2E-006 | 搜索资料 -> 保存资料卡 -> Agent 引用资料 | 资料卡包含 URL 和 retrievedAt。 |
| E2E-007 | 创建快照 -> 修改章节 -> 基于快照创建分支 | 主线和分支内容可区分。 |
| E2E-008 | 存在 pending revision 时导出 | 导出前有提示，默认不包含未接受内容。 |

## 3. 失败路径测试

| ID | 失败场景 | 预期 |
| --- | --- | --- |
| FAIL-001 | API Key 错误 | 显示认证失败，不创建空消息。 |
| FAIL-002 | LLM 流式中断 | AgentTask failed/cancelled，正文不损坏。 |
| FAIL-003 | 保存失败 | 编辑器保留内存态并提示重试。 |
| FAIL-004 | 导出路径不可写 | 显示错误，不改变项目。 |
| FAIL-005 | 搜索网页提取失败 | 保留搜索结果摘要并提示无法提取全文。 |
| FAIL-006 | 迁移失败 | 保留旧数据，提示备份路径。 |
| FAIL-007 | 修订应用时正文已变化 | 进入冲突状态，不强行覆盖。 |

## 4. 数据一致性测试

| 对象 | 测试点 |
| --- | --- |
| Project | 创建、重命名、删除、导出项目包后数据一致。 |
| Chapter | content、plainTextCache、wordCount 同步更新。 |
| Revision | pending/accepted/rejected/superseded 状态转换合法。 |
| Session | 消息顺序、流式消息、失败消息可恢复。 |
| Knowledge | 角色、设定、笔记的 projectId 与引用关系正确。 |
| Snapshot | 快照恢复或分支创建不丢失原数据。 |

## 5. 性能验收

| 场景 | 目标 |
| --- | --- |
| 打开 100 章项目 | 不一次性渲染所有正文，首屏可交互。 |
| 单章 2 万字 | 编辑、保存和字数统计无明显卡顿。 |
| Agent 流式输出 3000 字 | UI 可取消，滚动和输入不被长期阻塞。 |
| 全文搜索 500 条知识库记录 | 结果在可接受时间内返回。 |
| 导出 20 万字 Markdown/TXT | 不崩溃，进度可见。 |

## 6. LLM 生成代码验收提示模板

每次要求 LLM 实现模块时，建议附加：

```text
请基于现有领域模型实现【模块名】。
要求：
1. 不新增重复实体；如需字段，先说明迁移方案。
2. 所有写入通过 Repository 或 Command，不直接在 UI 中操作数据库。
3. 给出改动文件列表。
4. 给出正常路径、失败路径、边界路径测试。
5. 给出手动验收步骤。
6. 若有无法实现处，明确降级方案，不得静默跳过。
```
